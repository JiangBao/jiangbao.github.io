Node.js在游戏服务器的应用(一)

发表于 2017-07-11   |   分类于 后端技术

凭借着异步事件模型和V8的强势性能，Node.js在后端领域的应用相当活跃，在游戏领域的应用也是很有意义，给传统的游戏服务器开发带来了新的解决方案。虽然相比传统的C++、java类游戏服务器，可能还存在着性能调优、解决方案成熟性、健壮性等一些问题，但是Node.js的异步事件模型在I/O密集型的网络游戏应用中表现不俗，也带来了更加精简而快速的游戏开发体验。

简单记录自己游戏开发过程中应用Node.js的经验

关于游戏服务器

游戏服务器相比web服务器，在逻辑复杂性、通信量、实时性方面会有更高要求。一些常见区别：

  • web应用多为短连接，而游戏应用需要长连接保证实时性
  • 交互上：B/S更多关注某个用户和服务器交互，正确响应，游戏应用还会涉及很多广播消息，场景内消息同步等
  • 游戏应用连接状态更为复杂，应用长连接和网络的复杂性，需要考虑断线重连、恢复现场等复杂场景
  • 游戏玩家状态数据变化很频繁，缓存、数据持久化、安全检验设计需谨慎

几种常见的游戏服务器架构：

  1. 单进程服务器

    简单来说就是把socket服务、聊天服务、游戏逻辑等全塞到一个进程去处理，最简单甚至就可以用socket.io搭一个websocket服务器，接收到消息时丢给逻辑处理模块去处理。实现非常简单，但是承载用户量不大，实时性不高，后期拓展也很困难。
  2. 多进程服务器

    各个进程分工明确（承载连接、聊天、战斗等），进程间通过RPC或其它协议通信，各场景相互独立，对很对分区分服游戏是比较理想的方式。但场景服务器过载时动态拓展服务器比较困难。
  3. 分布式服务器

    和多进程服务器类似，各个进程服务器明确分工，进程间远程调用通信，但是服务器内部用master来集中管理所有服务器，实时观察负载状态，动态拓展性很强，但分布式带来的也是比较高的复杂度，部署在不同机子上的服务器进程状态同步、AOI设计等。所以还是应该根据产品需求有取舍地选择服务器架构，愉快做游戏开发。

主要包括三部分：

  • login server：登录认证服务器，主要负责登录验证，第三方渠道账号校验，使用express构建的http服务
  • game server： 游戏服核心，主要由前端服务、后端服务、master服务、数据层服务组成
    前端服务(连接层)由gate和connector组成，gate负责负载均衡，connector hold住所有连接，后端服务包括战斗服务器、邮件服务器、聊天服务器……master服务监控整个服务器群状态。
    基本的请求-响应流程：从登录服拿到服务器gate地址和登录token后，连上gate，gate返回合适的connector地址，客户端连上connector，connector会向login server做安全校验，校验成功则hold住连接，玩家绑定到相应channel，至此登录完成。客户端可以发起某个请求，connector收到请求后，根据自定义router将请求路由到相应服务器进程处理，后端服务器将处理结果返回connector，connector响应客户端，完成一次请求过程。
  • admin server：后台服务器，跟game server中master交互，获取服务器当前运行、负载状态，使用express构建的可视化管理工具。

这就是自己用Node.js构建游戏服务器的简单应用，抛开传统服务器的繁琐配置，开发体验比较愉快，但也有不少痛点，后面继续记录详细设计和踩过的坑
