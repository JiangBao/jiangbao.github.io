Node.js在游戏服务器的应用(二)

基础结构

上文说过，根据需求，将整个后端服务器群分为前端服务器(承载连接)和后端服务器(处理游戏逻辑)，各服务器都是单独进程，所以主要需处理的通信有两种：来自客户端的通信和服务器进程间的通信。在代码结构层面：所有服务器都放在servers目录下，每个服务器都包含handler、remote两部分，handler处理来自客户端的请求，remote处理进程间通信。基本代码结构如下：

servers中所有服务器通过handler暴露对外接口，收到请求，通过domain中的control层处理，数据层服务在dao 。客户端、服务端分离得很明确，服务端只提供json数据，客户端负责根据数据做渲染。前端请求以服务器名.handler名.方法名的方式添加路由，保证准确请求到相应服务。例如要请求connector的enter方法

当请求到达前端服务器，首先进行路由检查，如果前端服务定义了相应的handler，就在前端filter-handler链处理，否则，根据内部自定义router，得到相应后端serverId，发往后端服务器处理，后端处理完后返回前端服务器，再由前端连接层响应客户端。

服务端这样的分层设计很有意义，虽然牺牲了一些进程间通信带来的损耗，但带来了更高的拓展性和用户承载量。所以在经历过一个老旧的单进程服务器令人抓狂的维护期后，一直保持这样的架构，徘徊在多进程和分布式服务器之间。

连接模块

按照之前所说的设计，正常情况下，登录认证在connector层、本玩家库验证即可完成，曾经也这样做过，但由于游戏上线后接入多个渠道推广，各个渠道有各自的账户验证，connector层需处理的认证逻辑比较多，所以可以看到在上文的架构设计图中，将loginServer独立出去，用express搭一个http服务器，将各渠道登录请求路由到各渠道去认证。connector只需要跟loginServer做好token校验即可。

连上connector后即保持了长连接，两次心跳超时则认为客户端断线，为了良好的用户体验，客户端在检测到用户断线时，会在固定时间内主动发起断线重连。断线重连还是引入了不少技术复杂度的，常用的做法有：

  1. 断线即退出场景
    即认为玩家断线时就主动退出游戏场景，重连回来后只会回到游戏大厅，不会回到断线前的游戏场景。这是一种策划、前、后端开发都省事，但是对玩家很黑暗的做法，没有了场景恢复，前后端对玩家状态的处理会简单很多，但体验不会很好。
  2. 恢复现场
    即重连后恢复到玩家断线前的场景，并保持场景内玩家数据同步。服务器需要对玩家游戏状态(进入游戏、进入场景等)和场景状态(场景内各怪物数据)数据做好缓存，最初与客户端做场设计时要做好同步机制设计，在玩家断线发起重连请求时，根据缓存的玩家状态将玩家恢复到断线前场景，并将此时场景数据返回客户端同步。

由于网络环境复杂，重连涉及的玩家状态数据变化很多，所以经常会在断线重连这出现各种神奇的bug，算是比较不好解决的问题。

缓存机制

游戏服务器对玩家数据操作比较频繁，比如玩家金币、经验等数据，如果每次都直接操作持久化数据，从数据库读写，那实时性无法保证，所以缓存机制设计很有必要，大概思路是对于频繁改变的数据做缓存，再定时持久化数据，或按优先级分时持久化。读DB前，先读缓存，有则直接返回，没有再去读DB。主要考虑两点：

  1. 缓存数据的取舍
    一般游戏玩家数据会比较多(金币、经验等基础数据，邮件数据，任务数据…)如果将玩家所有数据都缓存，那太耗内存，所以应按照操作的频繁程度有取舍地缓存数据。如将玩家金币、经验等数据缓存，而邮件这样非频繁又庞大的数据直接落地。
  2. 缓存机制设计
    慎用进程内存做缓存，服务器宕掉缓存数据会丢失，尤其对于Node.js这种对内存使用有限制的(32位机器约0.7G，64位机器约1.4G)，直接使用进程内存缓存数据是比较危险的选择，一般可以使用memcache或者redis缓存数据，简单的key/value存储使用memcache，涉及存储、排序等复杂操作，redis是更好的选择。

之前项目中稳定使用的设计是：使用redis的hash表做缓存，持久化数据在mysql，服务器启动时从mysql读出需缓存数据，存到redis，redis主从设计，主库提供查询，从库与mysql同步数据。

异步流程控制

Node.js是单线程非阻塞的，代码层面表现为error-first的回调风格，大概执行流程是：同步任务在主线程执行，异步任务被塞入事件循环队列，同步任务执行完成后，再不断读取执行事件循环队列中的任务，即执行代码中的回调函数。所以当某一任务依赖前一任务执行结果，代码层就表现为需要在前一任务的回调函数中写这个任务，这样业务逻辑复杂时，就会形成多个回调函数的嵌套，代码就成了这样：

  • async库

  • 事件
    主要是使用node的events模块做一些发布/订阅的设计，比如曾经对任务模块做了这样的设计，游戏逻辑触发一些任务的解锁、完成等事件，任务订阅模块监听到事件后做后续处理，避免一些回调的使用。

    事件的正确触发时机和次数、触发后如何保证操作的原子性，都是使用事件来解决这些问题时需要考虑的，事件满天飞也会降低执行流的清晰度。

  • Promise
    异步编程的一种解决方案，将传统的回调嵌套改变为链式调用。promise主要概念：

    • 状态不受外界影响，主要包含Pending(进行中)、Resolved(完成)、Rejected(失败)三种状态
    • 状态一旦改变就不会再变，且状态改变只能有Pending --> Resolved和Pending --> Rejected两种

    ES6将Promise纳入规范，可直接使用，也可以使用Q库或者bluebird库，它们提供了更好的promise封装及操作方法。但不管使用哪种方式，都需要在原先异步代码中引入新的promise概念，并做一些封装，也就是promise化。

总得来说，不管使用哪种方式来控制异步流程，本质代码依旧是异步的，只是让维护者用更习惯的同步思维去阅读异步代码。不同人对回调的容忍程度也是不同的，比如自己曾经的团队维持了挺长时间的callback风格代码，所以对回调嵌套的容忍度高一些，反之，如果团队一直追求异步流程控制比较好的代码风格，那对回调嵌套则会十分厌恶。不管怎样，理解好异步的本质才是根本。
